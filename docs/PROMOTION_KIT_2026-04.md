# Multi Custom Agent Service — 프로모션 키트

## 1. 30초 소개

> Multi Custom Agent Service는 사내 문서와 조직 구조를 연결해, 여러 전문 챗봇이 협업 답변하는 엔터프라이즈 AI 플랫폼입니다. 단일 챗봇이 모든 질문을 억지로 처리하는 대신, Parent-Child 구조와 권한 정책을 통해 더 실무적인 응답 체계를 제공합니다.

---

## 2. 2분 소개

> 조직에는 이미 많은 지식이 있습니다. 문제는 그 지식이 문서, 위키, 메신저, 특정 담당자 기억 속에 흩어져 있다는 점입니다. Multi Custom Agent Service는 이 분산된 지식을 챗봇 구조로 재구성합니다.
>
> 사용자가 질문하면 Parent 챗봇이 먼저 해석하고, 필요하면 Child 전문 챗봇으로 위임합니다. 각 챗봇은 자신에게 연결된 문서 범위와 역할을 바탕으로 답변하며, 권한이 없는 사용자는 접근할 수 없습니다. 운영자는 관리자 화면과 JSON 설정을 통해 빠르게 챗봇을 관리할 수 있습니다.
>
> 결과적으로 이 플랫폼은 사내 지식 검색, 반복 문의 감소, 전문 팀 병목 완화, 권한 기반 안전한 AI 활용에 적합합니다.

---

## 3. 어떤 점이 중요한가

| 관점 | 메시지 |
|---|---|
| 업무 | 관련 팀 지식을 한 번의 질문으로 연결 |
| 기술 | Parent-Child 위임 구조 + RAG + 스트리밍 |
| 보안 | Knox ID 기반 권한 통제 |
| 운영 | JSON 기반 챗봇 정의, 관리자 UI 제공 |
| 확장성 | 팀/프로젝트별 챗봇을 점진적으로 추가 가능 |

---

## 4. 주요 기능

### 4-1. 멀티 챗봇 운영
- 챗봇별로 설명, 프롬프트, 문서 범위, 정책을 분리
- 도메인별/팀별 챗봇 운영 가능

### 4-2. 계층형 위임
- Parent가 질문을 1차 분류
- Child가 세부 도메인을 처리
- 필요 시 응답 종합 가능

### 4-3. 권한 제어
- 사용자별 챗봇 접근 권한 관리
- 접근 권한 없는 경우 명확한 오류 메시지 제공

### 4-4. 운영 가시성
- 위임 경로, 후보 점수, 출처 정보 확인 가능
- 잘못된 라우팅이나 권한 이슈 분석에 도움

---

## 5. 추천 사용 시나리오

### 시나리오 A — 사내 공통 허브
- Parent 챗봇이 여러 팀 Child 챗봇을 연결
- 사용자는 어떤 팀에 물어봐야 할지 몰라도 질문 가능

### 시나리오 B — 특정 팀 전문 챗봇
- 예: HR, DevOps, RTL, 프로젝트 운영
- 팀별 문서를 연결해 반복 질의를 줄임

### 시나리오 C — 프로젝트 전용 챗봇
- 프로젝트별 Parent/Child 구성
- 일정, 이슈, 모듈, 운영 정책을 구조적으로 분리 가능

## 6. 도입 시 기대 효과

### 정량 KPI는 파일럿으로 검증
아래 항목을 파일럿 단계에서 측정하는 방식이 적합합니다.
- 반복 문의 감소율
- 응답 시간 단축
- 사용자 재사용률
- 만족도/도움됨 평가
- 전문 담당자 문의 집중도 완화

### 정성적 효과
- "누구에게 물어봐야 하지?"를 줄일 수 있음
- 조직 구조를 반영한 응답 체계 구성 가능
- AI 도입 시 가장 어려운 운영/권한 영역을 함께 검토 가능

## 7. 데모 포인트

- 권한 없는 챗봇 접근 차단 시나리오

## 8. 자주 받을 질문

### Q. 일반 챗봇과 뭐가 다른가?
A. 이 서비스는 단일 챗봇이 아니라, 여러 전문 챗봇이 역할을 나눠 협업하는 구조입니다.

### Q. 운영이 복잡하지 않나?
A. 초기 설계는 필요하지만, 챗봇 정의와 정책이 JSON 중심이라 운영 체계가 명확합니다.

### Q. 보안은 어떻게 관리하나?
A. Knox ID 기반 사용자-챗봇 권한 모델을 사용하며, 접근 통제와 오류 메시지를 분리해 관리합니다.

### Q. 성능은 어떤가?
A. 질문 유형, 위임 구조, 하위 챗봇 수에 따라 달라지므로 파일럿 단계에서 최적 정책을 정하는 것이 좋습니다.
