사내 문서와 조직 구조를 반영해, 전문 챗봇들이 협업 답변하는 엔터프라이즈 AI 플랫폼
Parent → Child질문 성격에 따라 계층형 위임
Knox ID사용자-챗봇 권한 통제
JSON 운영신규 챗봇 추가/수정이 빠름

왜 필요한가

문제는 문서의 부재가 아니라 분산과 연결 부재다. 정보는 여러 시스템과 담당자에게 흩어져 있고, 질문은 반복되며, 답변은 일관되지 않다.

현업 문제플랫폼 접근
전문가 병목전문 Child 챗봇으로 역할 분산
정보 사일로Parent가 관련 도메인을 연결
운영 복잡도JSON + 관리자 UI로 구조 관리
보안 우려권한 없는 챗봇 접근 차단

핵심 가치

🧭

정확한 라우팅

Parent가 질문을 해석하고 적절한 Child를 선택하거나 직접 답변한다.

🔐

권한 기반 운영

Knox ID와 사용자-챗봇 권한 모델을 통해 안전하게 운영한다.

🧩

구조적 확장

팀/프로젝트별 챗봇을 Parent-Child 계층으로 확장할 수 있다.

⚙️

운영 가시성

위임 경로, 후보 점수, 출처를 확인하며 정책을 조정할 수 있다.

동작 구조

[사용자 질문] ↓ [Parent 챗봇] ├─ 직접 답변 가능 → 바로 응답 └─ 전문 영역 필요 → [Child 후보 선택] ↓ [문서 검색/RAG] ↓ [필요 시 다중 응답 종합] ↓ [최종 답변 + 출처]

대표 사용 시나리오

시나리오설명
공통 허브형Parent가 여러 팀 Child를 연결해 사용자가 질문 대상을 몰라도 응답 가능
전문 팀형HR, DevOps, RTL 등 도메인별 전문 챗봇으로 반복 질의 자동화
프로젝트형프로젝트 Parent 아래 모듈/업무별 Child를 두어 운영 정보 구조화
권한 통제형사용자별 챗봇 접근 허용/차단으로 안전한 내부 서비스 운영

파일럿에서 확인할 KPI

반복 문의 감소자주 묻는 질문 자동 응답 비율
응답 시간 단축질문 → 답변까지 걸리는 시간
사용자 만족도도움됨 평가 / 재사용률
운영 안정성권한 오류, 오선택, 위임 정책 개선 건수

추천 도입 방식

단계권장 내용
1단계FAQ가 많고 문서가 정리된 팀을 파일럿 대상으로 선정
2단계Parent/Child 구조와 권한 정책을 설계
3단계데모/시범 운영으로 위임 품질과 운영성 확인
4단계성과 확인 후 팀 단위로 확장
참고: 실제 정량 수치는 조직별 파일럿 결과로 산정하는 방식이 가장 안전하다.
