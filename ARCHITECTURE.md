# Multi-Tenant RAG Chatbot Platform
## Architecture Specification
**Version:** 1.2  
**Purpose:** System Regeneration Blueprint  
**최초 작성:** 2026-03-24

---

## 1. Purpose

이 문서는 멀티 챗봇 Retrieval-Augmented Generation 시스템의 핵심 아키텍처 원칙과 필수 요구사항을 정의한다.

**이 문서의 목적:**
- 시스템을 다시 생성할 수 있도록 하는 것
- 특정 구현체에 종속되지 않도록 하는 것
- AI 또는 개발자가 더 나은 구조를 자유롭게 설계할 수 있도록 하는 것
- 반드시 유지되어야 하는 아키텍처 원칙만 남기는 것

**이 문서는 의도적으로 다음을 포함하지 않는다:**
- 기존 프로젝트의 폴더 구조 / 파일명 / 코드 내용
- 특정 프레임워크 또는 라이브러리 강제
- 구현 세부사항에 대한 불필요한 제약

---

## 2. System Overview

하나의 플랫폼에서 여러 개의 독립 챗봇을 운영하는 **멀티 테넌트 RAG 시스템**.

각 챗봇은 독립적으로 다음을 가진다:
- 식별자
- 검색 대상
- 프롬프트/정책
- 모델 설정
- 대화 메모리 정책
- 실행 역할 (Tool / Agent)

> **핵심 정의:**  
> 챗봇은 사용자가 임의로 생성하는 대상이 아니라, **관리자가 선언형 설정(JSON 등)을 통해 등록/수정/삭제하는 대상**이다.  
> 새 챗봇 추가/제거는 **코드 복사 방식이 아닌, 선언형 정의의 변경만으로** 가능해야 한다.

## 3. Core Architectural Principles

### 3.1 Multi-Tenant Isolation
각 챗봇은 완전히 독립적인 실행 단위.

**반드시 보장:**
- 챗봇 간 검색 데이터 혼합 금지
- 챗봇 간 메모리 혼합 금지
- 허용되지 않은 데이터 소스 접근 금지
- 세션 상태의 교차 참조 금지

---

### 3.2 Explicit State Management
모든 상태는 명시적으로 관리되어야 한다.

- 숨겨진 내부 상태 객체에 의존 금지
- 대화 상태와 메모리는 **외부에서 식별 가능한 키**로 조회/저장 가능해야 함
- 대화 메모리는 최소한 `(chatbot_identifier, session_identifier)` 기준으로 식별 가능해야 함

---

### 3.3 Factory Method-Based Runtime Context Creation
**Factory Method 패턴**을 사용하여 챗봇별 실행 컨텍스트를 생성.

> 생성되는 것은 챗봇 정의 자체가 아니라,  
> 이미 등록된 챗봇 정의를 바탕으로 한 **런타임 실행 컨텍스트**이다.

**Factory Method 사용 이유:**
- 챗봇별 설정 차이를 실행 시점에 캡슐화
- 실행 컨텍스트 생성 표준화
- 새 챗봇 추가 시 코드 복사 없이 확장 가능
- 검색 / 메모리 / 모델 / 정책 / 실행 역할을 하나의 생성 흐름으로 묶기

**라이프사이클 분리:**
```
관리자 → 챗봇 등록/수정/삭제 (선언형)
요청 발생 → Factory Method → 런타임 실행 컨텍스트 생성
```

**Factory Method 책임 범위:**
- 챗봇 정의 해석
- 허용된 검색 대상 결정
- 메모리 네임스페이스 결정
- 모델 또는 응답 정책 결정
- 실행 역할(Tool/Agent)에 맞는 오케스트레이션 조합

> 구체적 클래스명, 모듈 구조, 파일 구조는 고정하지 않는다.

---

### 3.4 Manager-Oriented Resource Control
생성과 운영의 책임을 분리.

| 계층 | 책임 |
|------|------|
| Factory Method | 실행 컨텍스트 생성 |
| Manager 계층 | 챗봇 정의 / 검색 대상 / 세션 / 메모리 / 정책 등 운영 자원 관리 |

**달성 목표:**
- 생성 책임과 운영 책임의 분리
- 정책 변경 시 영향 범위 축소
- 멀티 챗봇 확장성 확보
- 향후 외부 저장소/인프라 연동 용이성 확보

> Manager의 구체적 형태는 자유롭게 설계 가능.  
> 단, 운영 자원에 대한 **통제 지점이 명확**해야 한다.

---

### 3.5 Declarative Chatbot Registration
챗봇은 관리자에 의해 **선언형 방식**으로 등록/수정/삭제.

**필수 요구사항:**
- JSON 등 기계가 읽을 수 있는 형식으로 정의 가능
- 코드 복사 없이 챗봇 추가 가능
- 비활성화/삭제가 쉬워야 함
- 설정 검증이 가능해야 함
- 런타임은 등록된 정의를 기준으로만 동작

---

### 3.6 Selectable Execution Role
각 챗봇은 실행 역할을 선택 가능해야 한다.

**최소 지원 역할:** `Tool` / `Agent`

> 이는 단순한 UI 옵션이 아니라,  
> **동일한 지식 경계를 가진 기능을 어떤 방식으로 외부에 노출하고 실행할지**를 결정하는 상위 개념이다.

**역할 선택 단위 (최소 하나 이상):**
- 챗봇 정의 단위
- 배포 단위
- 노출 인터페이스 단위

> 가능하다면 동일한 챗봇 정의를 서로 다른 역할로 동시에 노출할 수 있어도 좋다.

### 3.7 Hierarchical Chatbot with Session-Level Role Override

상위 챗봇은 하위 챗봇을 Tool 또는 Agent 역할로 사용할 수 있다.

**구조:**
- 챗봇은 하위 챗봇 목록을 레벨(level) 단위로 가질 수 있다
- 각 하위 챗봇 연결에는 **기본 실행 역할(default_role)** 이 정의된다
- 세션 시작 시 기본값을 **오버라이드(override)** 할 수 있다
- 세션마다 독립적인 역할 설정이 적용된다

**레벨 선택:**
- 하위 챗봇은 레벨(level)로 구분된다
- **최대 depth: 3 level** (level 1 / 2 / 3)
- 세션 생성 시 어느 레벨까지 활성화할지 선택 가능하다

**Chatbot Definition JSON 예시:**
```json
{
  "id": "chatbot-a",
  "name": "메인 챗봇",
  "sub_chatbots": [
    {
      "id": "chatbot-b",
      "level": 1,
      "default_role": "tool"
    },
    {
      "id": "chatbot-c",
      "level": 2,
      "default_role": "agent"
    },
    {
      "id": "chatbot-d",
      "level": 3,
      "default_role": "tool"
    }
  ]
}
```

**세션 시작 시 오버라이드 예시:**
```json
{
  "chatbot_id": "chatbot-a",
  "session_id": "user-session-xyz",
  "sub_chatbot_overrides": {
    "chatbot-b": "agent",
    "chatbot-c": "tool"
  },
  "active_level": 1
}
```

**격리 원칙 유지:**
- 역할이 달라져도 각 챗봇의 메모리/검색 격리는 유지된다
- 세션별 오버라이드는 해당 세션에만 적용된다 (전역 정의 변경 없음)

## 4. Core Concepts

### 4.1 Chatbot
독립된 정책과 데이터 경계를 가지는 기능 단위.

**최소 구성요소:**
- 고유 식별자
- 허용된 검색 범위
- 응답 생성 정책
- 메모리 스코프
- 실행 역할

---

### 4.2 Chatbot Definition
관리자가 선언형으로 등록하는 챗봇 명세.

**포함 가능한 내용:**
- 챗봇 식별자
- 검색 가능한 데이터 소스 목록
- 프롬프트 또는 정책
- 모델 관련 설정
- 메모리 정책
- 실행 역할 (Tool / Agent)
- 활성화 여부

> 런타임은 이 정의를 바탕으로만 실행 컨텍스트를 생성해야 한다.

---

### 4.3 Session
사용자 또는 호출 주체와 특정 챗봇 사이의 **연속적인 상호작용 흐름**.

**특성:**
- 여러 요청에 걸친 상태 유지
- 대화 기록의 조회 기준
- 메모리 저장의 단위
- 반드시 **명시적인 식별자**를 가져야 함

---

### 4.4 Memory
대화 맥락을 유지하기 위한 저장 메커니즘.

**필수 요구사항:**
- 세션 단위 관리
- 챗봇 단위 격리
- 외부 저장소로 대체 가능
- 제한 시간 또는 수명 정책 적용 가능

---

### 4.5 Retrieval
외부 지식 소스에서 관련 정보를 검색하는 기능.

**필수 요구사항:**
- 하나 이상 데이터 소스 검색 가능
- 결과 병합 가능
- 관련성 기준 정렬 가능
- 내부 구현 비노출

---

### 4.6 Execution Role
동일한 챗봇 기능을 어떤 방식으로 사용하고 노출할지를 정의.

#### Tool Role
다른 시스템이나 에이전트가 호출하는 **명시적 기능 단위**.

- 입력/출력 계약이 명확해야 함
- 범위가 비교적 제한적이어야 함
- 외부 오케스트레이터가 호출하기 쉬워야 함
- 필요 시 메모리 사용 여부를 명시적으로 제어 가능

#### Agent Role
보다 자율적으로 동작하는 **대화 주체 또는 실행 주체**.

- 대화 흐름을 직접 관리할 수 있음
- 메모리를 적극적으로 활용할 수 있음
- 필요 시 다른 기능을 조합해 응답할 수 있음
- 사용자 상호작용 중심의 경험 제공

> 시스템은 최소한 이 두 역할 중 하나를 **명시적으로 선택**할 수 있어야 한다.

## 5. System Layers

시스템은 논리적으로 다음 계층으로 구성된다:

```
Administration Layer
Interaction Layer
Application Layer
Retrieval Infrastructure
Model Inference Layer
State Storage Layer
```

### 5.1 Administration Layer
관리자가 챗봇을 등록, 수정, 비활성화, 삭제하는 계층.

**역할:**
- 챗봇 정의 관리
- 선언형 설정 검증
- 활성/비활성 상태 관리
- 실행 역할 및 정책 관리

> 이 계층은 챗봇의 생명주기를 관리하지만, 실제 요청 실행에는 직접 관여하지 않는다.

---

### 5.2 Interaction Layer
외부 호출이 들어오는 접점.

**예:** 웹 UI / API Client / Workflow System / 외부 Agent Framework

**역할:**
- 호출 수신
- 챗봇 식별자 전달
- 세션 식별자 전달 또는 유지
- 결과 수신 및 표시

> 이 계층은 검색 대상이나 내부 메모리 정책을 직접 결정하지 않는다.

---

### 5.3 Application Layer
플랫폼의 핵심 오케스트레이션 계층.

**역할:**
- 챗봇 정의 조회
- Factory Method를 통한 실행 컨텍스트 생성
- 세션 식별 및 복원
- 메모리 조회/저장
- 검색 호출
- 응답 생성 조율
- 실행 역할에 맞는 처리 흐름 선택
- 결과 반환

---

### 5.4 Retrieval Infrastructure
외부 검색 시스템 또는 검색 서비스 계층.

**역할:**
- 질의 수신
- 다중 데이터 소스 검색
- 검색 결과 반환

> Application Layer는 이 계층과 **API 계약을 통해 통신**해야 한다.

---

### 5.5 Model Inference Layer
LLM 또는 응답 생성 엔진 담당.

**역할:**
- 검색 결과와 대화 맥락을 바탕으로 응답 생성
- 필요 시 스트리밍 응답 지원
- 역할(Tool/Agent)에 맞는 응답 형식 지원

---

### 5.6 State Storage Layer
세션 및 메모리를 저장하는 계층.

**만족 요건:**
- 키 기반 조회/저장
- 챗봇/세션 단위 격리
- 교체 가능성
- TTL 또는 영속 저장 전략 선택 가능

---

## 6. Lifecycle Model

| 라이프사이클 | 설명 |
|-------------|------|
| 챗봇 등록 라이프사이클 | 관리자가 챗봇을 정의/등록하는 흐름 |
| 요청 실행 라이프사이클 | 실제 요청이 들어왔을 때의 실행 흐름 |

> 이 둘은 반드시 구분되어야 한다.

### 6.1 Chatbot Registration Lifecycle

챗봇은 런타임 호출 과정에서 생성되는 것이 아니라, **관리자가 미리 등록하는 대상**이다.

**개념적 흐름:**
1. 관리자가 새 챗봇 정의를 선언형 형식(JSON 등)으로 작성
2. 시스템이 정의의 유효성을 검증
3. 정의가 등록되면 챗봇은 활성 또는 비활성 상태를 가짐
4. 관리자는 정의를 수정하여 정책, 검색 대상, 실행 역할 등을 변경 가능
5. 관리자는 챗봇을 비활성화하거나 삭제 가능

**핵심:**
- 챗봇 자체는 **관리되는 리소스**다
- 런타임은 등록된 챗봇 정의를 소비한다
- Factory Method는 이 단계에서 챗봇을 "등록"하지 않고, 이후 실행을 위해 컨텍스트를 생성한다

---

### 6.2 Request Execution Lifecycle

1. 호출 주체가 특정 챗봇을 대상으로 요청을 보냄
2. 시스템이 등록된 챗봇 정의를 조회
3. Factory Method가 해당 정의를 바탕으로 실행 컨텍스트를 생성
4. 시스템이 세션 식별자를 확인하거나 생성
5. 필요한 경우 기존 메모리 맥락을 조회
6. 검색 인프라에 질의를 전달
7. 관련 문서를 검색 및 병합
8. 실행 역할(Tool 또는 Agent)에 맞게 응답을 생성
9. 응답을 반환
10. 필요한 경우 대화 기록 또는 실행 결과를 메모리에 반영

---

## 7. Retrieval Contract Requirements

Application Layer ↔ Retrieval Infrastructure 사이의 명확한 계약.

**검색 요청 최소 포함 내용:**
- 검색 대상 데이터 소스 집합
- 사용자 또는 호출 주체의 질의
- 반환 개수 또는 검색 범위

**검색 응답 최소 포함 내용:**
- 문서 또는 청크 내용
- 관련성 정보
- 출처 정보

> 응답은 반드시 **직렬화 가능**해야 하며, 상위 계층이 후처리 가능한 형태여야 한다.

## 8. Memory Requirements

### 8.1 Isolation
- 챗봇별 분리
- 세션별 분리

### 8.2 Replaceability
- 인메모리 저장소에서 시작 가능
- Redis, 데이터베이스, 분산 캐시 등으로 교체 가능해야 함

### 8.3 Lifecycle Control
- TTL 또는 만료 정책 지원 가능
- 장기적으로 요약/압축 전략 추가 가능

### 8.4 Server-Centric Control
- 메모리 정책은 서버가 관리
- 클라이언트는 메모리 내부 동작을 알 필요 없음

### 8.5 Role-Aware Memory Policy

| 역할 | 메모리 정책 |
|------|------------|
| Agent | 대화 연속성을 위한 메모리 사용이 기본 |
| Tool | 메모리 사용 여부가 명시적이고 제어 가능해야 함 |

> 역할 차이는 있어도 **메모리 키의 격리 원칙은 깨지면 안 됨**

---

## 9. Streaming Response Principle

**스트리밍 설계 시 유지 원칙:**
- 부분 응답 전송 가능
- 최종 응답 완료 후 메모리 반영 가능
- 출처 정보는 별도 구조로 전달 가능
- 실패 시 오류 처리가 가능해야 함

> 스트리밍 방식의 세부 구현은 자유다.

---

## 10. Execution Role Requirements

### 10.1 Tool Mode Requirements
- 호출 인터페이스가 명확해야 함
- 입력과 출력이 기계적으로 다루기 쉬워야 함
- 외부 오케스트레이터가 예측 가능하게 사용할 수 있어야 함
- 불필요한 UI 종속성이 없어야 함
- 필요 시 세션 및 메모리 사용이 명시적으로 제어 가능해야 함

### 10.2 Agent Mode Requirements
- 사용자 또는 상위 시스템과의 연속적 상호작용을 다룰 수 있어야 함
- 메모리를 활용한 문맥 유지가 가능해야 함
- 더 풍부한 응답 정책을 가질 수 있어야 함
- 필요 시 도구 활용 또는 다단계 응답 생성이 가능해야 함

### 10.3 Role Selection Model
- 실행 역할은 명시적이어야 함
- 역할 선택은 선언형 설정에 반영 가능해야 함
- 역할 선택은 런타임 오케스트레이션에 영향을 줄 수 있어야 함
- 역할이 달라져도 검색 격리와 메모리 격리는 유지되어야 함

---

## 11. Integration Requirements

**필요 조건:**
- 명시적인 API
- 직렬화 가능한 응답
- 세션 식별자 전달 가능성
- 챗봇 식별자 기반 실행 컨텍스트 생성 가능성
- 실행 역할(Tool/Agent)에 따라 다른 노출 방식을 허용하는 구조

> 향후 Tool 기반 AI 플랫폼 또는 Agent Framework 연동을 위한 요구사항.

---

## 12. Non-Functional Requirements

### 12.1 Scalability
- 챗봇 수 / 사용자 수 / 세션 수 / 검색 데이터 수

### 12.2 Reliability
고려해야 할 실패 상황:
- 검색 인프라 실패
- 모델 응답 실패
- 메모리 저장 실패
- 부분 응답 중단
- 잘못된 챗봇 정의 등록

### 12.3 Observability
추적 가능해야 하는 항목:
- 전체 요청 지연 시간
- 검색 시간
- 응답 생성 시간
- 챗봇별 사용량
- 역할별 호출 패턴
- 오류 발생 지점

## 15. Summary

**핵심 목표:**
- 하나의 플랫폼에서 여러 챗봇 운영
- 챗봇 간 완전한 격리
- 검색 기반 응답 생성
- 세션 기반 메모리 유지
- 외부 검색 인프라 연동
- 관리자가 선언형 설정으로 챗봇을 등록/관리
- 챗봇을 Tool 또는 Agent로 선택적으로 사용 가능

**시스템이 세워져야 하는 아키텍처 원칙:**
1. 관리자에 의한 선언형 챗봇 등록
2. Factory Method를 통한 런타임 실행 컨텍스트 생성
3. Manager 계층을 통한 운영 자원 통제
4. 명시적 식별자 기반 상태 관리
5. 교체 가능한 메모리 및 검색 인프라
6. Tool/Agent 실행 역할의 선택 가능성
7. 구현 세부사항에 종속되지 않는 확장성
